# CI/CDと自動テスト 導入・運用ガイド

本ドキュメントは、本プロジェクトにおけるCI/CD（継続的インテグレーション / 継続的デリバリー）パイプラインの導入手順、動作原理、および今後の発展（IaC・クラウド展開）に関する詳細な技術ガイドです。

## 1. 導入の目的
手動テストによる人為的ミスを防ぎ、コードの品質を自動で担保するためです。
特にIoTデータのような大量かつ異常値が混ざりやすいシステムでは、APIの入り口でのバリデーション（Pydanticによる型チェックや異常値の遮断）が確実に機能しているかを機械的に検証する（TDD: テスト駆動開発）ことが、システムダウンを防ぐ上で不可欠です。

## 2. 導入された技術スタック
- **テストフレームワーク**: `pytest`, `httpx` (FastAPI標準の `TestClient` と連携)
- **CIプラットフォーム**: GitHub Actions (構成ファイルを用いたインフラ自動構築の基礎)

---

## 3. 実装の詳細と導入方法

### 3.1. テストコード (`test_main.py`) の実装
FastAPIの `TestClient` を使用し、実際にサーバーを立ち上げることなくエンドポイントのテストを実行します。

- **正常系**: 期待されるJSONデータが正しく処理され、ステータスコード `200` が返却されるか。
- **異常系・セキュリティ**: Pydanticの `Field` 制約（絶対零度以下の温度など）や `extra="forbid"`（未定義・未知のプロパティの拒否）が正確に作動し、`422 Unprocessable Entity` が返却されるか。

**【ローカルでの実行方法】**
仮想環境を有効化した後、以下のコマンドでテストを実行します。
```bash
pytest test_main.py -v
```

### 3.2. GitHub Actions (`.github/workflows/ci.yml`) の設定
リポジトリの `.github/workflows/` ディレクトリにYAMLファイルを配置するだけで、GitHubが自動的にCIパイプラインを構築・実行します。
この「設定ファイル（コード）でサーバーの構築手順を記述する」手法は、モダンな **Infrastructure as Code (IaC)** の基礎的な考え方です。

**【ワークフローの処理フロー】**
1. **`runs-on: ubuntu-latest`**: クラウド上に最新のUbuntu仮想マシンをプロビジョニング（確保）。
2. **`actions/checkout@v4`**: 起動した空のUbuntu内に、現在のリポジトリのソースコードを自動でクローン。
3. **`actions/setup-python@v5`**: Python 3.13環境を瞬時に構築し、システムパスを通す。
4. **依存関係のインストールとテスト**: `pip install` でライブラリを導入し、`pytest` を実行。

これにより、「開発者のローカル環境に依存しない、まっさらなクリーンな環境」でテストが成功することが保証されます。

---

## 4. 今後の発展展望（CDとIaCの本格導入）

現在のシステムは「Pushされたコードのテストを自動化する（CI）」という第一段階をクリアしています。
これを実際のBtoBビジネスや本番環境に適応させるための「次のステップ」は以下の通りです。

### ① Dockerによるコンテナ化
開発環境と本番環境の差異を完全になくすため、`Dockerfile` を作成し、FastAPIアプリケーションをコンテナ化します。これにより、環境構築の手間がさらに省けます。

### ② クラウド（Amazon ECS）への自動デプロイ (CD: 継続的デリバリー)
GitHub Actionsのワークフローを拡張し、`pytest` が成功した場合のみ、作成したDockerコンテナイメージをAmazon ECR（コンテナレジストリ）に保存し、自動的に **Amazon ECS (Fargate)** へデプロイされる仕組みを構築します。
これにより、「コードをPushしてテストが通れば、数分後には本番環境に新機能が安全にリリースされる」というエンタープライズ水準の完全なパイプラインが完成します。

### ③ Terraform / AWS CDKによるインフラのコード化 (IaCの極意)
AWSのネットワーク設定（VPC）やサーバーのスペック変更など、AWSコンソール画面から手作業でポチポチと行う構築作業を、すべて **Terraform** や **AWS CDK** といったツールを用いてコード（テキストファイル）で定義します。
今回の `ci.yml` で「Ubuntuの構築をコード化した」のと同じ思想で、**「AWSインフラの構築自体をコード化してバージョン管理する」** ことで、より高度な IaC (Infrastructure as Code) スキルを持つエンジニアとして活躍することができます。
