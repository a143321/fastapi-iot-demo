# GitHub Actions OIDC Migration Guide

本プロジェクトでは、CI/CDパイプライン（GitHub Actions）からAWS環境（ECRなど）へのデプロイにおける認証方式として、初期の「IAMユーザー（アクセスキー方式）」から、よりセキュアな **「OIDC (OpenID Connect) 連携方式」** へと移行（マイグレーション）しています。

このドキュメントでは、両者のアプローチの違いと、OIDCを採用するアーキテクチャ上のメリットについて解説します。

## 🛡️ 認証方式の比較

| 比較項目 | ❌ IAMユーザー（アクセスキー方式） | ✅ OIDC連携（一時クレデンシャル方式） |
| :--- | :--- | :--- |
| **認証の仕組み** | 発行された静的な文字列（アクセスキーとシークレット）を使用 | クラウドプロバイダ（GitHubとAWS）間の信頼関係に基づく一時トークンを使用 |
| **認証情報の寿命** | **永続的**（手動で無効化・ローテーションするまで有効） | **一時的**（ジョブの実行中のみ有効。発行後数十分で自動消滅） |
| **漏洩時のリスク** | **非常に高い**（ソースコードやログからキーが漏れた場合、悪意のある第三者にAWSを不正利用される） | **極めて低い**（トークンが漏れてもすぐに無効化され、特定のリポジトリからしか発行されない） |
| **シークレット管理** | **煩雑**（GitHub Secretsに永続的なパスワードを登録・管理する必要がある） | **不要**（IAMロールのARNを指定するだけでよく、パスワード管理から解放される） |
| **セキュリティ評価** | 非推奨（レガシーな手法であり、AWS IAMのベストプラクティスに反する） | **推奨（モダンなクラウドネイティブアーキテクチャの標準仕様）** |

## 🏗️ OIDC連携の仕組み（アーキテクチャ）

OIDCを利用した認証フローは以下の通りです。

1. **信頼関係の構築**: あらかじめAWS側のIAMロール（`github-actions-oidc-role`）に、「対象のGitHubリポジトリ（`a143321/fastapi-iot-demo`）の `main` ブランチからのリクエストのみを信頼する」というポリシーを設定しておきます。
2. **トークン要求**: GitHub Actionsがジョブを実行する際、GitHubのOIDCプロバイダに対して一時的なJWT（JSON Web Token）を要求します。
3. **AWSへの提示**: 取得したJWTをAWS STS（Security Token Service）に提示し、一時的な認証情報（セッショントークン）を要求します。
4. **検証と許可**: AWSはJWTを検証し、信頼されたリポジトリからの要求であることを確認した上で、数分間だけ有効な一時クレデンシャルを発行します。
5. **リソースへのアクセス**: GitHub Actionsは、この一時クレデンシャルを使ってECRへのDockerイメージプッシュなどの操作を行います。

## 💡 導入によるビジネスバリュー

本プロジェクトにおいてOIDCを採用することは、単なる技術的なアップデートにとどまらず、エンタープライズ環境で求められる **「ゼロトラストセキュリティ」** および **「クレデンシャル漏洩リスクの根本的排除」** を実現するものです。これにより、安全かつ運用コストの低い継続的デリバリー（CD）環境が構築されています。
